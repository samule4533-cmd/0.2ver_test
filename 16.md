## 개념 정리 — RAG 파이프라인 전체 흐름

### 데이터 흐름

```
PDF
  → pdf_parser.py (Gemini File API → Markdown)
  → chunker.py (헤더 기반 분리 → 청크 리스트)
  → output_writer.py (vector_chunks.json 저장)
  → company_vectordb.py (ChromaDB upsert)
  → [여기까지 완료]

질문 입력
  → rag_chain.py
      query_collection()  ← ChromaDB에서 유사 청크 검색
      → 프롬프트 조립 (청크 본문 + 메타데이터)
      → Gemini LLM 호출
      → 답변 + 출처(chunk_id, header, source_file) 반환
```

### 핵심 개념 정리

| 항목 | 내용 |
|------|------|
| `vector_chunks.json` 역할 | 적재 시에만 사용. 이후 RAG는 ChromaDB를 직접 조회 |
| 코사인 유사도(distance) | 검색 필터 역할 — 거리값이 높은(관련 없는) 청크를 LLM에 넘기기 전 제외 |
| 코사인 유사도 ≠ 답변 품질 | 유사도는 검색 단계 필터, 실제 답변 품질은 end-to-end RAG 테스트로 평가 |
| end-to-end 답변 | 질문 → 검색 → 프롬프트 조립 → LLM 호출 → 자연어 답변(출처 포함) |
| case별 테스트 타이밍 | `rag_chain.py` 구현 완료 후 진행 — 지금 단계에서는 불필요 |
